### 5.4 住所自動補完（郵便番号 → 番地サジェスト）

郵便番号入力から番地選択までをシームレスに行う直列連携方式。

#### フロー

```
① ユーザーが郵便番号7桁を入力（例: 120-0036）
    ↓ 自動発火
② zipcloud API で都道府県・市区町村・町域名を取得
   → 「東京都足立区千住仲町」を住所欄に自動入力
    ↓ 自動発火
③ 自動入力された文字列を Google Places Autocomplete に渡す
   → 「千住仲町19-3」「千住仲町20-1」等の番地候補をドロップダウン表示
    ↓ ユーザーが選択
④ 番地まで確定。建物名・部屋番号欄（任意）のみ手入力
```

#### UI構成

```
┌──────────────────────────────────────┐
│ 郵便番号    [120 - 0036]             │  ← 7桁入力で②③が自動発動
│ ご住所      [東京都足立区千住仲町  ▼] │  ← 番地候補がドロップダウン
│              19-3                    │
│              19-5                    │
│              20-1                    │
│ 建物名等    [○○マンション201]        │  ← 任意・手入力
└──────────────────────────────────────┘
```

#### 使用API

**① zipcloud（郵便番号 → 町域名）**

- エンドポイント: `https://zipcloud.ibsnet.co.jp/api/search?zipcode={7桁}`
- 無料・APIキー不要・CORS対応
- レスポンス例:
```json
{
  "results": [{
    "zipcode": "1200036",
    "address1": "東京都",
    "address2": "足立区",
    "address3": "千住仲町"
  }]
}
```
- クライアントサイドから直接fetch可能

**② Google Places Autocomplete（町域名 → 番地サジェスト）**

- ライブラリ: `@googlemaps/js-api-loader` または `<script>` 直接読み込み
- 無料枠: Essentials tierで月10,000リクエスト無料（今回の規模なら十分）
- セッショントークンを使用してAutocomplete → Place Detailsをまとめると課金最適化
- API制限設定:
  - `componentRestrictions: { country: 'jp' }` で日本国内に限定
  - `types: ['address']` で住所のみにフィルタ
- 実装例:
```typescript
const autocomplete = new google.maps.places.Autocomplete(inputElement, {
  componentRestrictions: { country: 'jp' },
  types: ['address'],
  fields: ['formatted_address', 'address_components']
});

// zipcloudの結果をinputに自動入力した後、focusでサジェスト発動
autocomplete.addListener('place_changed', () => {
  const place = autocomplete.getPlace();
  // place.formatted_address → 完全な住所を取得
  // place.address_components → 都道府県、市区町村、番地を個別取得可能
});
```

#### 環境変数（追加分）

Google Maps Embed APIと同じプロジェクト・APIキーで運用可能。ただしAPIキーの制限設定で **Places API（New）** も有効化が必要。

```env
# Google Maps（Embed + Places Autocomplete 共用）
NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=  # Embed APIキーと統合。Places APIも有効化すること
```

## 10. API設計

### 10.1 エンドポイント

#### `POST /api/submit`

フォーム送信処理。

**リクエスト**: `multipart/form-data`

| フィールド | 型 | 必須 | 備考 |
|-----------|---|------|------|
| name | string | ✅ | ご芳名 |
| attendance | string[] | ✅ | Notion Multi-selectの値をそのまま送信 |
| postal_code | string | 条件付き | ハイフン付き（例: `120-0036`） |
| address | string | 条件付き | 番地まで含む完全住所 |
| building | string | — | 建物名・部屋番号 |
| relation | string | — | 陽介とのご関係 |
| nickname | string | — | 陽介から呼ばれていたあだ名 |
| message | string | — | 陽介へのメッセージ（長文可） |
| photos | File[] | — | 画像ファイル（複数可） |

**処理フロー**:
```
1. バリデーション
   - name: 必須チェック
   - attendance: 1つ以上選択チェック
   - postal_code / address: attendance に参列系が含まれる場合は必須
   - photos: MIMEタイプチェック（image/*のみ許可）

2. 写真アップロード（並列処理）
   - 各ファイルを Cloudflare R2 にアップロード
   - キー: photos/{timestamp}-{uuid}-{originalFilename}
   - 返却: 公開URLの配列

3. Notion DBにページ作成
   - POST https://api.notion.com/v1/pages
   - ヘッダー: Authorization: Bearer {NOTION_API_KEY}, Notion-Version: 2022-06-28
   - ご住所: address + building を結合して格納
   - 写真URL: R2公開URLを改行区切り（\n）で結合して rich_text に格納
   - 送信日: サーバー側でISO 8601形式の現在日時を設定

4. Discord Webhook通知
   - POST {DISCORD_WEBHOOK_URL}
   - Content-Type: application/json

5. レスポンス返却
```

**レスポンス**: `200 OK`

  "showLineQR": true

**エラーレスポンス**: `400 Bad Request` / `500 Internal Server Error`

## 13. セキュリティ・プライバシー

- 自宅住所はLP上に掲載しない。フォームで「自宅線香」を選択した方にのみLINE友達追加導線を表示
- 管理画面はパスワード保護
- R2バケットはパブリック読み取り可（写真URL直接アクセス）だが、URLを知らない限りアクセスできない（リスト表示は無効化）
- Notion APIキー等の秘匿情報はサーバーサイドのみで使用
- フォーム送信にはレート制限を設けることを推奨（1IP/1分に1回程度）

## 14. パフォーマンス要件

- LCP（Largest Contentful Paint）: 2.5秒以内
- ヒーロー写真は next/image で最適化（WebP変換、レスポンシブ）
- フォント: Google Fonts の `display=swap` で表示ブロック回避
- 写真アップロード: クライアントサイドでリサイズ（長辺2000px程度に縮小）してからR2に送信し、転送量を削減

## 15. 開発・運用メモ

### 15.1 写真差し替え

ヒーロー写真・About写真は `/public/photos/` に配置。ファイル名を固定（`hero.jpg`, `about-1.jpg`, `about-2.jpg`）にしておけば、画像ファイルの差し替えだけでデプロイ不要（Vercel ISRまたはビルド時に反映）。

### 15.2 フォームクローズ

告別式後にフォームを閉じたい場合:
- 環境変数 `NEXT_PUBLIC_FORM_CLOSED=true` を設定
- フォーム部分を「受付は終了しました」の表示に切り替え

### 15.3 Notion DBの事前準備

開発開始前に以下を準備:
1. Notionで新規データベースを作成（プロパティは6.1に準拠）
2. Notion Integrationを作成してAPIキーを取得
3. データベースにIntegrationを接続

### 15.4 Cloudflare R2の事前準備

1. Cloudflareアカウントでバケット `yosuke-kiddy` を作成
2. R2 API Tokenを発行（Object Read & Write権限）
3. パブリックアクセスを有効化（カスタムドメイン or r2.dev サブドメイン）

### 15.5 Discord Webhookの事前準備

1. 通知を受け取りたいDiscordチャンネルで Webhook URL を発行

### 15.6 ドメイン・DNS設定

- ドメイン: `yosuke-kiddy.com`（Cloudflare Registrarで取得済み）
- Vercelプロジェクト Settings → Domains で `yosuke-kiddy.com` を追加
- Cloudflare DNS に以下を追加:
  - タイプ: `CNAME` / 名前: `@` / ターゲット: `cname.vercel-dns.com` / プロキシ: **オフ（DNSのみ・グレー雲）**
- SSL: Vercel側で自動発行（Cloudflareプロキシは必ずオフ）

## 16. Google API 設定手順

### 16.1 Google Cloud Console 設定

1. Google Cloud Console → プロジェクト作成（または既存を使用）
2. 以下のAPIを有効化:
   - **Maps Embed API**（式場地図の埋め込み用）
   - **Places API (New)**（住所オートコンプリート用）
3. APIキーを1つ作成（両APIで共用）
4. APIキーの制限:
   - HTTP リファラー制限: `https://yosuke-kiddy.com/*` のみ許可
   - API制限: Maps Embed API, Places API (New) の2つのみ許可

### 16.2 Maps Embed（式場地図）

埋め込みURL形式:
```
https://www.google.com/maps/embed/v1/place
  ?key={NEXT_PUBLIC_GOOGLE_MAPS_API_KEY}
  &q=稲垣葬儀社+稲垣ホール
  &center=35.7497,139.8048
  &zoom=16
```

### 16.3 Places Autocomplete（住所サジェスト）

- ライブラリ読み込み: `@googlemaps/js-api-loader` を使用
- 無料枠: Essentials tierで月10,000リクエスト無料
- セッショントークンを使用して課金最適化
- 詳細な実装仕様はセクション5.4を参照
